启动新代理来自主处理复杂的多步骤任务。

Task 工具在隔离会话中生成一个专门的子代理，拥有独立的上下文窗口。主动使用——不要等用户要求。

可用的代理类型及其可访问的工具：
{agents}

使用 Task 工具时，你必须指定 subagent_type 参数来选择代理类型。

## 何时使用 Task 工具

**探索与研究（explore 代理）：**
- 在修改前理解不熟悉的代码
- 查找函数、类型或模式在代码库中的所有用法
- 追踪代码路径、依赖关系或导入链
- 回答"X 是如何工作的？"或"X 在哪里定义？"
- 通过搜索相关代码和错误模式来调查 bug
- 映射模块或子系统的架构

**实现与执行（general 代理）：**
- 跨多个文件或模块进行更改
- 实现跨越多个代码区域的功能
- 实现更改后编写测试
- 重构涉及多处的代码
- 任何可以独立运行的多步骤工作

## 何时不使用 Task 工具
- 读取单个已知文件路径 → 直接使用 Read
- 搜索特定的类/函数名 → 直接使用 Glob 或 Grep
- 在 1-3 个特定文件中搜索 → 直接使用 Read
- 不需要研究即可在 1-2 个工具调用中完成的任务

## 代理选择

| 场景 | 代理 | 原因 |
|------|------|------|
| "X 是如何工作的？" | explore | 只读、快速、专注于查找和阅读 |
| "找到所有匹配...的文件" | explore | 专门用于 glob/grep/read |
| "...的架构是什么？" | explore（非常彻底） | 深度探索，追踪完整依赖链 |
| "实现功能 X" | general | 可以读取和写入，处理多步骤工作 |
| "跨文件重构 X" | general | 需要编辑多个文件 |
| "为 X 编写测试" | general | 需要读取代码并写入测试文件 |
| "调查 bug X" | 先 explore | 研究 bug，然后直接修复或通过 general |

## 并行执行

在任务独立时，在单条消息中并行启动 2-3 个代理（多个工具调用）。这是最大化性能和完整性的主要方式。

分解模式：
- "添加功能 X" → explore(找类似代码) + explore(找相关测试) → 然后 general(实现)
- "修复 Y 中的 bug" → explore(追踪 bug) + explore(检查 Y 的测试覆盖)
- "重构 Z" → explore(找所有使用处) + explore(找约定) → general(区域 A) + general(区域 B)
- "理解模块 M" → explore(M 的 API 和结构) + explore(M 的依赖和消费者)

默认并行。仅在 agent2 依赖 agent1 的结果时顺序执行。

## 编写有效的提示词

每个代理从你的对话中获得零上下文。你的提示必须自包含：

1. **陈述目标**：代理应该确切地完成什么？
2. **提供起点**：文件路径、函数名、搜索模式
3. **指定范围**：写代码还是仅研究？关注哪些目录？
4. **定义输出**：回复应包含什么信息？
5. **对于 explore**：指定彻底程度——"quick"、"medium" 或 "very thorough"
6. **对于 general**：包含验证步骤（测试命令、预期行为）

差："看一下认证模块"
好："非常彻底地探索认证系统，从 src/auth/ 开始。找到：1) 端到端登录流程，2) 会话令牌存储，3) 认证中间件链。返回每个关键函数的文件路径和行号。"

## 工作树隔离（默认）

所有任务默认在隔离的 git worktree 中运行（`isolation="worktree"`）。这意味着：
- 每个子代理在自己的仓库副本上工作——并行代理之间不会干扰
- 子代理完成后，更改通过 `git apply --3way` 自动合并回主工作目录
- 如果自动合并成功，输出包含 `<worktree_merge status="applied">`
- 如果自动合并失败（冲突），输出包含原始 diff 供手动解决

对于只读任务（explore 代理），设置 `isolation="none"` 可以避免约 2 秒的 worktree 设置开销。
