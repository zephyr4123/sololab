Launch a new agent to handle complex, multistep tasks autonomously.

The Task tool spawns a specialized subagent in an isolated session with its own context window. Use it proactively — do not wait for the user to ask.

Available agent types and the tools they have access to:
{agents}

When using the Task tool, you MUST specify a subagent_type parameter to select which agent type to use.

## When to use the Task tool

**Exploration & Research (explore agent):**
- Understanding unfamiliar code before making changes
- Finding all usages of a function, type, or pattern across the codebase
- Tracing code paths, dependencies, or import chains
- Answering "how does X work?" or "where is X defined?" questions
- Investigating bugs by searching for related code and error patterns
- Mapping architecture of a module or subsystem

**Implementation & Execution (general agent):**
- Making changes across multiple files or modules
- Implementing features that span several areas of the codebase
- Writing tests after implementing changes
- Refactoring code that touches many locations
- Any multi-step work that can run independently

**Slash Commands:**
- Executing custom slash commands. Pass the command invocation as the entire prompt.
  Example: Task(description="Check file", prompt="/check-file path/to/file.py")

## When NOT to use the Task tool
- Reading a single known file path → use Read directly
- Searching for a specific class/function name → use Glob or Grep directly
- Searching within 1-3 specific files → use Read directly
- Tasks completable in 1-2 tool calls without research

## Agent Selection

| Scenario | Agent | Why |
|----------|-------|-----|
| "How does X work?" | explore | Read-only, fast, focused on finding and reading |
| "Find all files matching..." | explore | Specialized for glob/grep/read |
| "What's the architecture of..." | explore (very thorough) | Deep exploration, traces full dependency chain |
| "Implement feature X" | general | Can read AND write, handles multi-step work |
| "Refactor X across files" | general | Needs to edit multiple files |
| "Write tests for X" | general | Needs to read code and write test files |
| "Investigate bug X" | explore first | Research the bug, then fix directly or via general |

## Parallel Execution

Launch 2-3 agents IN PARALLEL (single message, multiple tool calls) whenever their tasks are independent. This is the primary way to maximize performance and thoroughness.

Decomposition patterns:
- "Add feature X" → explore(find similar code) + explore(find related tests) → then general(implement)
- "Fix bug in Y" → explore(trace the bug) + explore(check test coverage for Y)
- "Refactor Z" → explore(find all usage sites) + explore(find conventions) → general(area A) + general(area B)
- "Understand module M" → explore(M's API and structure) + explore(M's dependencies and consumers)

Default to PARALLEL. Only go sequential when agent2 depends on agent1's results.

## Writing Effective Prompts

Each agent starts with ZERO context from your conversation. Your prompt must be self-contained:

1. **State the goal**: What exactly should the agent accomplish?
2. **Provide starting points**: File paths, function names, search patterns
3. **Specify scope**: Write code or just research? Which directories to focus on?
4. **Define output**: What information should the response include?
5. **For explore**: Specify thoroughness — "quick", "medium", or "very thorough"
6. **For general**: Include verification steps (test commands, expected behavior)

Bad: "Look at the auth module"
Good: "Very thorough exploration of the auth system starting from src/auth/. Find: 1) Login flow end-to-end, 2) Session token storage, 3) Auth middleware chain. Return file paths with line numbers for each key function."

## Worktree Isolation (Default)

All tasks run in an **isolated git worktree** by default (`isolation="worktree"`). This means:
- Each sub-agent works on its own copy of the repository — no interference between parallel agents
- After the sub-agent finishes, changes are **automatically merged** back to the main working directory via `git apply --3way`
- If auto-merge succeeds, the output includes `<worktree_merge status="applied">`
- If auto-merge fails (conflict), the output includes the raw diff for manual resolution
- The worktree is cleaned up automatically after merge

Set `isolation="none"` ONLY for read-only tasks (explore agent) when you want to avoid the ~2s worktree setup overhead.

## Additional Features

- **Session resumption**: The output includes a `task_id`. Pass it back to continue the same agent session with its full context preserved.

## Examples

<example>
user: "Add pagination to the users API endpoint"
<plan>
1. Need to understand current API structure and patterns → launch 2 explore agents in parallel
2. Then implement based on findings
</plan>
assistant: Launches two Task tools in a single message:
- Task(subagent_type="explore", description="Find API endpoint patterns", prompt="Medium exploration of the API layer. Find: 1) How existing endpoints handle pagination (search for 'offset', 'limit', 'page' patterns), 2) The users endpoint implementation, 3) Related request/response types. Start from src/api/ and src/routes/. Return file paths and key code patterns found.")
- Task(subagent_type="explore", description="Find API test patterns", prompt="Quick exploration: find existing API test files, especially any that test pagination. Search in tests/ directory. Return file paths and test patterns used.")
</example>

<example>
user: "Refactor the logger to use structured JSON output"
<plan>
1. Explore: find all logger usages across the codebase
2. Implement: apply changes per area in parallel
</plan>
assistant: First launches explore:
- Task(subagent_type="explore", description="Find all logger usages", prompt="Very thorough exploration: find every file that imports or uses the logger. Start from src/lib/logger and trace all consumers. Also find the logger's configuration and initialization. Return a complete list of files and the specific logger patterns each uses.")
Then, based on results, launches multiple general agents:
- Task(subagent_type="general", description="Refactor logger core", prompt="Refactor src/lib/logger.ts to output structured JSON. [details from explore results]... Run 'npm test -- logger' to verify.")
- Task(subagent_type="general", description="Update logger consumers", prompt="Update these files to use the new structured logger API: [file list from explore]... Ensure existing log levels are preserved.")
</example>
