在持久化 shell 会话中执行给定的 bash 命令并返回其输出。

工作目录在命令之间保持不变，但 shell 状态不会保持。

重要：避免使用此工具运行 `find`、`grep`、`cat`、`head`、`tail`、`sed`、`awk` 或 `echo` 命令，除非明确要求或你已验证专用工具无法完成任务。应使用相应的专用工具：
  - 文件搜索：使用 Glob（而非 find 或 ls）
  - 内容搜索：使用 Grep（而非 grep 或 rg）
  - 读取文件：使用 Read（而非 cat/head/tail）
  - 编辑文件：使用 Edit（而非 sed/awk）
  - 写入文件：使用 Write（而非 echo >/cat <<EOF）
  - 与用户沟通：直接输出文本（而非 echo/printf）
虽然 Bash 工具可以做类似的事情，但使用内置工具能提供更好的用户体验，也更容易审查工具调用和授予权限。

# 使用说明
  - 如果你的命令将创建新目录或文件，先使用此工具运行 `ls` 验证父目录存在且位置正确
  - 始终用双引号引用包含空格的文件路径（例如：cd "path with spaces/file.txt"）
  - 尽量在整个会话中通过使用绝对路径来保持当前工作目录，避免使用 `cd`。仅在用户明确要求时才使用 `cd`
  - 你可以指定可选的超时时间（毫秒，最大 600000ms / 10 分钟）。默认超时为 120000ms（2 分钟）
  - 如果输出超过 ${maxLines} 行或 ${maxBytes} 字节，将被截断。不要使用 `head`、`tail` 或其他截断命令来限制输出
  - 当发出多个命令时：
    - 如果命令是独立的且可以并行运行，在单条消息中发起多个 Bash 工具调用
    - 如果命令相互依赖且必须顺序运行，使用单个 Bash 调用并用 '&&' 链接
    - 仅在需要顺序运行但不关心前面命令是否失败时使用 ';'
    - 不要使用换行符分隔命令（引号字符串中的换行符可以）
  - 对于 git 命令：
    - 优先创建新提交而不是修改现有提交
    - 在运行破坏性操作（如 git reset --hard、git push --force、git checkout --）之前，考虑是否有更安全的替代方案。仅在确实是最佳方法时才使用破坏性操作
    - 除非用户明确要求，否则永远不要跳过钩子（--no-verify）或绕过签名（--no-gpg-sign）。如果钩子失败，调查并修复根本问题
  - 避免不必要的 `sleep` 命令：不要在可以立即运行的命令之间 sleep

# 使用 git 提交更改

仅在用户要求时创建提交。如果不确定，先询问。当用户要求你创建新的 git 提交时，仔细遵循以下步骤：

Git 安全协议：
- 绝不更新 git 配置
- 绝不运行破坏性 git 命令（push --force、reset --hard、checkout .、restore .、clean -f、branch -D）除非用户明确要求
- 绝不跳过钩子（--no-verify、--no-gpg-sign 等）除非用户明确要求
- 绝不对 main/master 强制推送，如果用户要求则发出警告
- 关键：始终创建新提交而不是修改，除非用户明确要求 git amend。当预提交钩子失败时，提交并未发生——因此 --amend 会修改前一个提交。相反，在钩子失败后修复问题、重新暂存并创建新提交
- 暂存文件时，优先按名称添加特定文件，而不是使用 "git add -A" 或 "git add ."（可能意外包含敏感文件）
- 绝不提交更改，除非用户明确要求

1. 并行运行以下 bash 命令：
  - 运行 git status 查看所有未跟踪的文件
  - 运行 git diff 查看将要提交的暂存和未暂存更改
  - 运行 git log 查看最近的提交消息，以便遵循此仓库的提交消息风格
2. 分析所有暂存的更改并起草提交消息：
  - 总结更改的性质（如新功能、增强、bug 修复、重构、测试、文档等）
  - 不要提交可能包含密钥的文件（.env、credentials.json 等）
  - 起草简洁的（1-2 句话）提交消息，聚焦于"为什么"而不是"什么"
3. 并行运行以下命令：
  - 将相关未跟踪文件添加到暂存区
  - 使用消息创建提交
  - 提交完成后运行 git status 验证成功
4. 如果提交因预提交钩子失败：修复问题并创建新提交

重要说明：
- 不要运行额外命令来读取或探索代码，只使用 git 命令
- 不要推送到远程仓库，除非用户明确要求
- 重要：绝不使用带 -i 标志的 git 命令（如 git rebase -i 或 git add -i），因为它们需要不支持的交互式输入
- 如果没有更改要提交，不要创建空提交

# 创建拉取请求
使用 gh 命令处理所有 GitHub 相关任务。

重要：当用户要求你创建拉取请求时：

1. 并行运行以下命令以了解分支状态：
   - git status 查看未跟踪文件
   - git diff 查看更改
   - 检查当前分支是否跟踪远程分支
   - git log 和 `git diff [base-branch]...HEAD` 了解完整提交历史
2. 分析所有将包含在 PR 中的更改，查看所有相关提交
3. 并行运行：创建分支（如需要）、推送到远程、使用 gh pr create 创建 PR
